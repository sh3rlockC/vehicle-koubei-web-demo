# 前端界面重规划

基于 `frontend-design` skill 的要求，本 demo 不继续沿用通用深色面板风格，而是明确转向“车型口碑情报舱”：一个给部门内部使用的双平台采集、AI 归纳和老板汇报工作台。核心记忆点是：用户不是在填表，而是在启动一次双源口碑侦察任务。

## 设计定位

- 目标用户：部门内部同事，需求是快速输入车型、确认平台车系、等待采集、下载结果、向 AI 追问。
- 产品语气：工业情报台、克制但有压迫感，强调任务链路、双 agent 并行、数据产物和 AI 洞察。
- 差异化记忆点：进度页做成“双采集雷达”，汽车之家和懂车帝两条采集线从 0 到 100 并行推进，后续摘要、词云、一页纸和问答像流水线接力。
- 视觉约束：不使用紫色白底、Inter/Roboto/Arial 默认风格、普通后台表单布局；保留中文可读性和内网/云端部署稳定性。

## 视觉系统

- 色彩：主背景使用近黑蓝 `#07111d`，主面板使用深墨蓝 `#101b2a`，信号蓝 `#73d0ff` 代表采集进度，工程绿 `#89f7b8` 代表成功，警戒橙 `#ffbf69` 代表降级/重试，故障红 `#ff7d7d` 代表失败。
- 字体：标题使用更有机械感的中文黑体栈，例如 `"DIN Alternate", "Avenir Next Condensed", "Noto Sans SC"`；正文使用 `"Noto Sans SC", "PingFang SC"`，避免默认系统栈裸奔。
- 背景：全局加入蓝图网格、低透明扫描线、局部径向光斑，形成“控制室屏幕”氛围。
- 卡片：面板不再只是玻璃拟态圆角，而是加入细描边、切角、状态灯、编号铭牌，让每个区域像仪表模块。
- 动效：页面进入时采用从左到右的任务链路显现；进度条加入扫描高光；候选卡选择时有短促信号闪烁；结果页回答区域使用自然展开，不截断文本。
- 响应式：桌面端强调左右分区和双采集并行；移动端改为纵向任务卡，下载和继续按钮固定在底部可触达区域。

## 全局信息架构

- 顶部改为“任务指挥条”：显示当前步骤、车型、任务 ID、访问状态、当前运行阶段。
- 左侧或顶部保留 5 步路线：口令、车型、车系确认、任务执行、结果洞察。
- 每页主标题改为业务语言，例如“启动口碑侦察”“锁定平台车系”“双源采集中”“生成洞察包”。
- 所有页面都保留“重新开始”和“返回上一步”的明确入口，避免任务中断后用户迷路。
- 错误态不只显示错误文本，还给出可执行动作：重试、手动输入车系 ID、返回车型输入、查看后台状态。

## 页面规划

### 1. 口令页

- 页面角色：进入情报舱的门禁，不是普通登录页。
- 主视觉：中央“访问口令终端”，右侧显示本周口令版本、访问说明、无账号登录说明。
- 交互：输入框强化大号数字/字符输入体验，错误态明确提示“口令不正确或已过期”。
- 设计细节：口令通过后出现一条短暂的“授权完成”扫描动画，再进入车型页。

### 2. 车型输入页

- 页面角色：任务启动台。
- 主区域：一个极简但强聚焦的车型输入框，支持示例车型快捷填入。
- 辅助区域：显示“系统将执行什么”：识别汽车之家车系、识别懂车帝车系、双 agent 采集、生成 ZIP 和 AI 洞察。
- 风险提示：说明外网查找可能耗时，已确认过的车型将优先走缓存或本地结果。
- 后续接口预留：可展示最近成功任务、常用车型、缓存命中状态。

### 3. 候选确认页

- 页面角色：双平台车系锁定。
- 布局：汽车之家和懂车帝左右双列，每列只突出当前推荐候选，其他候选折叠为“更多候选”。
- 信息展示：候选标题、seriesId、来源 URL、匹配原因、是否人工输入。
- 手动兜底：改为“高级校正”抽屉，不默认占据主流程空间。
- 关键规则：两个平台必须都锁定后，底部主按钮才变为“启动双源采集”。

### 4. 任务进度页

- 页面角色：整个 demo 的核心展示页。
- 双采集雷达：汽车之家和懂车帝两张大卡并列，分别显示 agent_id、当前状态、采集页数、已采评论数、进度百分比、最近消息。
- 并行感：两条采集线必须在视觉上同时出现，即使其中一个等待中，也要显示“已投递/等待 agent 响应”，避免用户误以为串行。
- 下游流水线：采集完成后，后处理、摘要、词云、一页纸、问答索引用横向流水线显示。
- 状态解释：重试、降级、失败需要有明确颜色和一句人话说明。
- 后台操作预留：管理员后续可在此页增加“取消任务”“刷新状态”“复制 job_id”。

### 5. 结果页

- 页面角色：从“产物列表”升级为“洞察交付台”。
- 第一屏：顶部直接给出车型、样本量、完成状态、下载全部结果 ZIP 的主按钮。
- 智能一页纸：作为首个内容区，使用报告式排版，包含一句结论、老板摘要、关键机会点、风险提醒。
- 词云与 Excel：不再分散为多个下载项，只保留一个“下载全部结果 ZIP”按钮；页面内可以预览词云缩略图和样本概览。
- 智能问答：改为全宽对话/报告区，回答文本使用自然换行和自适应高度，不使用固定高度、不截断、不显示省略号。
- 问答辅助：保留追问建议，但按钮放在回答下方，避免抢占主要回答区域。

## 组件拆分建议

- `AppShell`：统一顶部任务指挥条、背景、页面宽度和路线导航。
- `StepRail`：5 步流程状态，支持 desktop 横向、mobile 纵向或压缩。
- `SignalPanel`：通用仪表卡片，支持状态灯、标题、辅助说明和右上角指标。
- `CandidateColumn`：平台候选选择卡，复用汽车之家和懂车帝。
- `CollectorLane`：单个平台采集进度卡，展示 agent_id、百分比、消息和状态。
- `PipelineStrip`：采集后的处理流水线。
- `InsightReport`：智能一页纸展示组件。
- `ResultBundleCard`：统一 ZIP 下载与产物说明。
- `QaConsole`：智能问答输入、回答和追问建议。

## 实施顺序

1. 先重构全局视觉系统：`globals.css` 变量、背景、按钮、卡片、输入框、状态色。
2. 增加共享组件：`AppShell`、`StepRail`、`SignalPanel`，减少每页重复布局。
3. 改口令页和车型输入页，先确定整体视觉方向和中文文案。
4. 改候选确认页，压缩手动兜底，把双平台确认做成主视觉。
5. 改任务进度页，把两个 agent 的采集进度做成核心展示。
6. 改结果页，突出 ZIP 下载、智能一页纸和完整问答显示。
7. 最后做移动端校验、无数据/失败/降级状态校验，以及本地浏览器跑通一次完整任务。

## 不建议现在做的事

- 不建议马上引入复杂 UI 框架，当前 Next.js 页面用 CSS 和少量组件即可支撑 demo。
- 不建议做登录系统，当前“链接 + 周口令”符合内部轻量使用方式。
- 不建议把微信小程序纳入同一轮前端重构，先把 Web 版跑顺后再抽 API 和视觉规范给小程序。
- 不建议把进度页做成纯日志流，双平台并行采集是 demo 的关键卖点，必须可视化。

## 验收标准

- 用户 10 秒内能理解这个工具做什么：输入车型、双平台采集、生成 AI 洞察包。
- 用户能明确看到汽车之家和懂车帝是两个采集 lane，并理解两者是否同步推进。
- 结果页第一屏能完成“看结论”和“下载 ZIP”两个动作。
- 智能问答回答完整显示，不出现文本截断或省略号。
- 页面在桌面和手机宽度下都能完成完整流程。
